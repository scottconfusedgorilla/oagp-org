# agent-sdk examples

## `run_demo.py` — Tier-1 propose-only autonomous dispatch

Generates the dispatch artifacts for a propose-only `doc-reviewer` bound agent
against `demo-orgdef.opencatalog` (a self-contained demo org with one embedded
roledef), using the `WorkflowsBackend`:

```bash
cd agent-sdk
PYTHONPATH=src python examples/run_demo.py s:/path/to/project/.claude/agents
```

It produces:
- the bound subagent file (`<agents_root>/doc-reviewer.md`) — `tools: Read, Write, Glob, Grep`, **no Bash** (the Tier-1 structural bound);
- a bind-event audit memo;
- `doc-reviewer.workflow.js` — the Claude Code workflow that dispatches the bound agent via `agentType`.

## The two dispatch paths (and why `agentType` is the real one)

`WorkflowsBackend` generates a workflow that dispatches via **`agentType`**, not an inline prompt. This is load-bearing: only `agentType` carries the bound agent file's `tools:` frontmatter, which is the Tier-1 hard layer. Dispatching the prompt inline would hand the agent the workflow's default broad toolset (including `Bash`) and silently break bounded authority.

`demo-output/` holds evidence from a 2026-05-29 run:
- `structural-agenttype-path.workflow.js` — the real path (generated by `WorkflowsBackend`).
- `mechanics-demo.workflow.js` — an inline-dispatch variant used only to demonstrate the read→work→file-proposal mechanics when `agentType` could not resolve in-session (see below). **Not** the structural path — inline cannot constrain tools.
- `review-proposal.md` — the propose-only output the dispatched agent actually produced (7 proposals; README left unedited).
- `bind-event-doc-reviewer.{openthing,body.md}` — the audit-trail memo `run_seat()` emitted for the dispatch.

## The session-start rule (empirical, load-bearing)

Running the structural `agentType` workflow **in the same session that wrote the agent file fails**:

```
agent type 'doc-reviewer' not found. Available agents: claude, claude-code-guide, Explore, general-purpose, Plan, statusline-setup
```

Claude Code resolves `agentType` against the registry loaded **at session start**. A subagent file written mid-session is not visible until a session that loads after it exists — exactly the constraint the 2026-05-23 bind() PoC hit (it resolved via Ctrl+N → new session).

**Operational consequence for autonomous dispatch:** `run_seat()` should be driven by a launcher that starts a clean session per dispatch (the agent file pre-exists that session). This composes naturally with the autonomous/cron use case — each dispatch is a fresh session — and is why the backend generates a runnable script rather than dispatching in-process.

## Tier-3 note for the Workflows path

The `OAGP_BOUND_AGENT` env marker cannot be injected into a workflow-spawned subagent. For the Workflows path the non-delegable-dispatch floor therefore rests on **package-absence** (do not expose `oagp_agent_sdk` in the dispatch environment) plus the bound toolset — a deployment guarantee, not an in-package one. Surfaced as a coordination finding to oagp-strategist.
